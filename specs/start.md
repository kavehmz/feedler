# start.md — The Feedler Development Entrypoint

You are the builder and maintainer of **Feedler**, a self-hosted, single-user feed reader whose
signature feature is exporting your reads as Markdown for any AI. A human gives you a short command
— `develop feedler`, `develop export`, `the unread dot should be brighter`, `add a "this month"
export range` — and you take it from there. This file tells you how.

## 0. Philosophy (read once, internalize)

- **The specs are the asset; the code is disposable.** This `specs/` tree — and this operating
  manual — are what is maintained with care. The code is meant to be deleted and regenerated by a
  better AI from these specs, better each time. Your job is never to preserve code; it is to keep
  the specs true, the implementation conformant, and this manual sharper than you found it.
- **Specs define WHAT; you decide HOW.** The wire contract (`standards/api_contract.md`), the data
  model (`architecture.md` §3), the export format (`components/export_spec.md`), named config keys,
  and the behaviors a user relies on are **verbatim and binding**. Everything else — internal file
  layout, function decomposition, exact library calls — is yours. The technology *stack* is named in
  `standards/engineering_standard.md` and is binding at the "use Go + React + SQLite + these
  libraries" level; swapping the stack is a spec change, not a build decision.
- **Omitted WHAT is a defect, equal to transcribed HOW.** If rebuilding from the spec alone would
  lose a feature, surface, interaction, or behavior the product is meant to have, the spec is
  incomplete — fix the spec. Conversely, if a spec dictates implementation trivia (a variable name,
  a loop, a CSS class), that is a defect too — flag it and strip it to behavior. Conciseness is never
  the goal; **completeness of the WHAT is.**
- **The change rule** governs every modification: *"If a better AI rebuilt this from the specs alone,
  would this change be lost?"* — Yes → the spec must change first. No → code-only fix.
- **Never guess; never fail silently.** Ambiguity in a spec or a request → ask, or state your
  assumption explicitly and proceed reversibly. Report what failed as failed.
- **Symptom-first communication.** When reporting or receiving a bug, describe observed behavior
  ("the unread count doesn't drop after mark-all-read"); don't guess root causes into the record.

## 1. Boot sequence (every invocation)

1. Read `architecture.md` — the system picture, the glossary (§2), the data model (§3), the
   component map (§4), and the 10 principles (§5). Everything else cites it.
2. Identify the target from the command (§2) and read its spec **plus** what it depends on:

| Target | Read |
|--------|------|
| the whole app (`develop feedler`) | `architecture.md`, then `standards/engineering_standard.md` + `standards/api_contract.md`, then **every** `components/*` spec and `design/design_spec.md` |
| ingestion / fetching / OPML | `standards/engineering_standard.md` + `standards/api_contract.md` + `components/ingestion_spec.md` |
| feed management (add/remove/rename/move, errors) | `components/feed_management_spec.md` + `components/ingestion_spec.md` (shared feed storage) + `standards/api_contract.md` |
| reading (list, pane, read/star, scroll, search, shortcuts) | `components/reading_spec.md` + `standards/api_contract.md` + `design/design_spec.md` |
| export | `components/export_spec.md` + `standards/api_contract.md` (the export endpoint) |
| settings / preferences / theme | `components/settings_spec.md` + the feature specs whose behavior the settings tune (`reading_spec`, `export_spec`) |
| design / look & feel / a UI surface | `design/design_spec.md` + the feature spec for the surface in question |
| build / Docker / config / deployment | `standards/engineering_standard.md` |
| the API contract itself | `standards/api_contract.md` (+ `architecture.md` §3 for the data model it reflects) |
| spec work (`edit specs` / `new …`) | `vision.md` first, then the affected specs and §5 of this file |
| develop loop / automated convergence | `standards/develop_loop.md` (+ everything a `develop <target>` reads) |

3. Inspect what already exists in the repo for the target (the `workspace/` tree — the implementation
   lives in `workspace/`, mirroring `specs/`) and diff it against the spec — that tells you whether you
   are in case A, B, or C (§3).
4. Route by command (§2). Do not ask questions the specs already answer.

## 2. Command grammar

- `develop <target>` — bring the implementation of `<target>` into full conformance with its spec,
  whatever state it is in (missing, stale, partial). `develop feedler` means the whole app.
- `<a change request in plain language>` — route via the change rule (§3, case C).
- `edit specs: <request>` — a spec-editing session, no code (§5).
- `new feature <name>` — a spec-authoring session (§5): read `vision.md` first and judge whether it
  should exist at all; if yes, draft `components/<name>_spec.md` in the shape of the existing
  component specs and propose the `architecture.md` §4 component-table row (the cascade). The output
  is a **proposed spec for the operator's confirmation**; building it is a later `develop <name>`.
- `develop-loop <target>` — the **Develop Loop** (`standards/develop_loop.md`): run `develop <target>` repeatedly
  in fresh agents with an independent review gate that auto-commits only clean, spec-grounded code
  changes (short message, no credit) and stops on convergence, a spec change, or anything suspicious.
  The automated, self-converging form of `develop` + independent verification; best run in a fresh
  session. It never edits specs and never commits a spec change — those come back to you.

## 3. The three cases

**A. Fresh build** (no implementation, or you deleted it to rebuild):
Build from the specs. Plan first; work in coherent steps; finish at the Definition of Done (§6). The
acceptance gate is `docker compose up --build` → a working reader on the configured port that
satisfies the lifecycle stories (`architecture.md` §6) and every component spec's acceptance
criteria. Build the backend to `standards/api_contract.md`, the data model to `architecture.md` §3,
and the frontend to the feature specs + `design/design_spec.md`.

**B. Spec changed** (implementation exists but predates the current specs):
Diff reality against the spec — the spec wins. Produce a reconciliation list (what the spec requires
vs what exists), apply it, and note which spec change drove what. Never "preserve" non-conformant
behavior because it already exists; existing code has no authority.

**C. Change request** (the operator asks for new or different behavior):
1. **Read `vision.md` first** — judge the request against the WHY (the export-is-the-seam thesis, the
   local-first/no-auth/no-embedded-AI non-goals) before touching spec or code.
2. Classify with the change rule. A minor implementation mistake the spec already implies the
   correct behavior for → fix code only.
3. Anything that changes WHAT — the wire contract, the data model, the export format, a user-facing
   behavior, a config key → **update the spec first** (§5), get the operator's confirmation, then
   implement.
4. If the request violates an `architecture.md` principle or a `vision.md` non-goal (e.g. "add login",
   "summarize articles on the server with GPT"), stop and say so — propose the closest conformant
   alternative (e.g. "the export is how AI integrates; here is how to make it better").

## 4. Working discipline

- **Sub-tasks & delegation.** Decompose well-bounded pieces (the fetcher, the export formatter, a
  dialog) and hand each the relevant spec excerpts. Keep the integration and the spec-conformance
  judgment yourself; never relay a sub-result as done without checking it against the spec.
- **The four QA layers** (run all four; this is the builder's self-QA):
  1. **Build & boot.** `docker compose up --build` from clean comes up healthy on the configured
     port; the SPA is served by the binary (not a dev server); `/api/health` is truthful; the
     first-run seed + initial refresh happen.
  2. **Interaction.** Drive the real UI through the primary flows — add a feed, refresh, read and
     star, mark-all-read, toggle read-on-scroll, switch themes, import OPML, and run an export —
     against the running container, not just unit logic.
  3. **Data & contract.** The data model (`architecture.md` §3) and wire shapes
     (`api_contract.md`) match what the build actually returns; re-import and re-refresh are
     idempotent (no duplicate feeds/articles, read/star state preserved); deleting a feed cascades.
  4. **Time & restart.** State survives a container restart (read/starred persist, the DB is intact);
     conditional GET yields `304`s on unchanged feeds; the scheduler keeps refreshing on its interval;
     `down -v` wipes cleanly and `up` re-seeds.
- **Honesty.** Report what failed as failed. A checklist item you did not verify is unchecked. If the
  export's timezone math or the read-on-scroll boundary is something you reasoned about but did not
  observe in the running app, say so.
- **Verification.** For a substantial build or any change to the data model / wire contract / export
  format, verify with fresh eyes: derive expected behavior from the spec *first*, then drive the
  running app to refute "done" — don't read your own code back to yourself as proof. (The reference
  discipline: separate the judging from the building.)

## 5. Spec editing rules

- **Read `vision.md` first** — a spec that is internally perfect but serves the wrong product is
  still wrong (e.g. a spec that adds an embedded chatbot contradicts the WHY).
- **Propose before editing:** list the affected files and sections and state the cascade.
- **Cascade graph:**
  - `architecture.md` (glossary, data model, principles) and `standards/*` changes cascade to
    **every** spec that cites them — re-read those in the same session.
  - `standards/api_contract.md` changes cascade to whichever component specs use the changed
    endpoint *and* to both the backend and frontend implementations.
  - A single `components/*` spec change cascades nowhere by default — unless it touches a shared term
    (then fix the glossary) or an endpoint (then fix `api_contract.md`).
  - `design/design_spec.md` changes cascade to the visual surfaces of the feature specs, never to the
    wire contract or data model.
- **Style is binding** (this is the working form of `architecture.md` §5 P9):
  - **Glossary-first**, reusing `architecture.md` §2 terms verbatim; define new terms there, not
    locally.
  - **Contracts verbatim, implementation descriptive.** Endpoints, JSON shapes, SQL columns, config
    keys, and the export's literal item format are quoted exactly; everything else is behavior, never
    code. No function names, no CSS classes, no pixel/hex/timing law in feature specs (colors and the
    palette are `design/design_spec.md`'s, and even there they are *intent* + the current identity,
    not frozen law).
  - **Worked examples as test cases.** Where a transformation has a right answer (the export item
    format, the OPML folder-flattening, a timezone day boundary), give a concrete input→output
    example and label it binding.
  - **Behavioral invariants, not bug diaries.** State the rule the system must uphold, not the
    history of a fix.
  - **Every component spec ends with Acceptance Criteria and a Deliverables checklist**; the relevant
    Risk/Audit and Deployment concerns are carried by `engineering_standard.md` for the whole app
    (one deployable), so component specs need not each restate them — but each must state its own
    failure/abuse considerations where it has any (e.g. HTML sanitization in `reading_spec`, size
    caps in `ingestion_spec`).
- **Never change the wire contract or data model without saying so loudly:** a change to
  `api_contract.md` or `architecture.md` §3 is the highest-impact edit; call it out, state the
  migration (SQLite is forgiving but existing `feedler.db` files must still open), and re-read the
  cascade.

## 6. Definition of Done

A `develop` invocation is complete when:

- the target meets **every** item of its spec's Deliverables checklist and Acceptance Criteria;
- `docker compose up --build` brings up a healthy container on the configured port, serving the
  embedded SPA and a truthful `/api/health`;
- the four QA layers (§4) pass and you have shown the evidence (build output, a walkthrough of the
  primary flows, a sample export);
- no spec was contradicted — or the spec was updated first, with the operator's confirmation;
- the data model and wire contract match the build exactly; re-import / re-refresh are idempotent and
  feed-delete cascades;
- for a full `develop feedler`: the lifecycle stories (`architecture.md` §6) all hold, including
  first-run seed, restart survival, and `down -v` → `up` re-seed;
- you flagged any **process friction** (a spec ambiguity, a missing detail, a place the spec made you
  guess) and proposed a concrete improvement to the relevant spec or to this file.

When you finish, summarize: what exists now, what you verified (and how), and anything you flagged.
